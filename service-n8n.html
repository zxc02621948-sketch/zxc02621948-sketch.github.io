內容

會做到什麼程度

流程設計

先把你現在的做法拆解成步驟，找出哪幾步可以自動、哪幾步該留給人決定。不是每個環節都適合自動化，該留的我會說。

API 與 Webhook 串接

串接你在用的服務：Google Sheets、資料庫、LINE、Email、各種有開 API 的系統。沒有 API 的，看情況用其他方式接。

失敗會被你知道

網路斷線、對方系統掛掉、資料格式不對，這些都會發生。流程會重試，重試還是失敗就發通知給你，不會默默死掉讓你三天後才發現。

可以自架，資料留在你這

n8n 可以架在你自己的主機上，資料不用經過第三方平台。我現在自己就是這樣跑的，也能幫你架。

流程文件

每個流程做什麼、串了哪些服務、失敗的時候去哪看，都會寫成文件給你。人事異動的時候接手的人看得懂。

加 AI 的部分

需要理解自由文字的環節可以接語言模型，例如把客戶來信分類、從信裡抽出訂單資訊。但金額、庫存這類數字我會用程式算，不交給模型。

做不到的事我先講：如果對方系統完全封閉、沒有 API 也不能匯出檔案，那自動化就沒有施力點。這種情況我會在評估階段就告訴你，不會收了錢才說做不了。
實績

我自己在跑的東西

自架 n8n 對外開放 API，前端網頁跨網域呼叫

Webhook · CORS · 語言模型串接
● 運作中
內容

把 n8n 流程包裝成一支對外的 API：網頁前端送出請求，流程接收後呼叫語言模型抽取資料，再用程式邏輯計算，最後把結構化結果回傳給網頁顯示。

為什麼

這種做法讓自動化流程變成一個可以被任何系統呼叫的服務，而不是關在後台自己跑。要換前端、要多接一個系統，流程本身不用動。

多租戶 AI 客服的自動化骨架

狀態管理 · 防禦性解析 · 真人接手 · 多通道
實際營運中
內容

整套客服系統跑在 n8n 上：訊息進來、判斷客人在流程的哪一段、呼叫模型、驗證模型回傳的格式、失敗就走備援分支、信心不足就轉真人。多個客戶共用一套系統，各自的設定與知識庫分開。

重點

每一次模型回傳都會經過格式驗證跟重試，避免它偶爾吐出奇怪的東西就把整個流程弄壞。做自動化最花工夫的通常是這些防呆，不是主流程。
