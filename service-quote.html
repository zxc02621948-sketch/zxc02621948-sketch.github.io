核心做法

AI 判斷「是什麼」，程式決定「多少錢」

文字理解與數值計算分開處理，方便檢查每個品項與金額的來源。

文字理解

從自由文字中整理品項、數量與特殊條件，並輸出固定欄位供後續比對。

目錄比對與計算

依品項比對目錄，再計算數量、工時、運費、稅額與利潤；結果保留對應的目錄項目與計算規則。

這樣拆分的原因：語言模型可能辨識錯品項，不適合自行產生價格。品項先與目錄比對，價格再由既定規則計算；找不到對應資料時標記為待確認，不自動填入估計值。
適合的狀況

這些報價流程適合導入

  • 報價要人工翻目錄業務或技師寫一段需求，行政人員得逐項查價、算稅、打成單子。一份報價要花掉半天。
  • 不同人報出來的價格不一樣同樣的需求，看誰處理，最後金額可能差一截。折扣、工時、運費各憑經驗。
  • 收到的文件格式不固定客戶傳來的規格書、需求單、詢價信每份長得都不一樣，要人讀完才知道要什麼。
  • 試過讓 AI 直接報價，但結果無法核對品項或價格可能不在目錄中，也看不出數字依據。

海外客戶
問題

技師到現場後用自由文字描述要做什麼，行政人員得逐項翻三家供應商的目錄查價，再算利潤、工時、車馬費跟稅，最後打成報價單給客戶。很花時間，也常出錯。

做法

語言模型從法文需求中擷取品項與數量，程式再比對供應商目錄並依規則計價。找不到的項目標為「待確認」，不填入估計值；結果包含利潤、工時費、車馬費與增值稅，可轉為 PDF。

結果

單價與計算結果可追溯到供應商目錄和計價規則，不由模型自行產生。流程部署在自架 n8n，並以 API 供前端網頁呼叫，已用實際施工需求測試。

其他用途

相同做法也可用於整理詢價信、從規格書擷取欄位，或建立訂單草稿；實際比對方式仍需依你的資料與規則設計。

內容

交付內容

  • 欄位擷取定義需要從文字中取得的欄位，並驗證模型輸出格式；格式不符時重試或標記待處理。
  • 比對規則將擷取出的品項對應到目錄，並處理同義詞、別名與規格差異。
  • 缺漏標記無法對應目錄的項目集中列出，由人確認後再完成報價。
  • 計價規則將利潤、工時費、運費、稅率與折扣條件寫成固定規則，集中管理與修改。
  • 結果輸出可產生 PDF 報價單、試算表，或透過 API 傳送到既有系統。
資料品質會影響辨識結果：若同一品項有多種名稱、價格分散在不同檔案，需先整理目錄或建立對照規則。報價前會先抽樣測試，再估算資料整理與開發範圍。
