AI 報價 / 文件處理

起價 NT$40,000

有人用一段話描述需求，系統把它變成一張可以出去的報價單。數字全部來自你的目錄，模型只負責看懂那段話。

核心做法

AI 判斷「是什麼」，程式決定「多少錢」

這是這類系統唯一重要的設計決定，也是我在每個案子裡都堅持的分工。

交給模型的部分

讀懂一段自由文字，判斷裡面提到哪些品項、各多少數量、有沒有特殊條件。這是模型擅長的事，換個講法它也認得出來。

交給程式的部分

拿著品項去比對你的目錄，抓單價、算數量、加工時、加運費、算稅、套利潤。每一個數字都有出處，追得到是目錄裡的哪一筆。

為什麼要這樣拆：如果讓模型自己報價，它會給你一個看起來很合理的數字，而且看起來合理的錯誤最難被發現。拆開之後，模型答錯只會是「品項認錯」，你一眼看得出來；價格則永遠是對的，因為它根本不是模型算的。目錄裡找不到的東西會被標成待確認，留給人處理，不會被塞一個估計值進去。
適合的狀況

如果你有這些困擾

  • 報價要人工翻目錄業務或技師寫一段需求，行政人員得逐項查價、算稅、打成單子。一份報價要花掉半天。
  • 不同人報出來的價格不一樣同樣的需求，看誰處理，最後金額可能差一截。折扣、工時、運費各憑經驗。
  • 收到的文件格式不固定客戶傳來的規格書、需求單、詢價信每份長得都不一樣，要人讀完才知道要什麼。
  • 試過讓 AI 報價，但不敢用它會編造出目錄裡根本沒有的品項跟價格，而且講得很有自信。
案例

瑞士水電工程的 AI 報價助手

技師用法文寫一段施工需求，系統輸出完整報價單

法文自由文字 · 供應商目錄比對 · PDF 產出 · 自架 n8n API
海外客戶
問題

技師到現場後用自由文字描述要做什麼，行政人員得逐項翻三家供應商的目錄查價，再算利潤、工時、車馬費跟稅，最後打成報價單給客戶。很花時間，也常出錯。

做法

那段法文進來，由語言模型抽出品項與數量，接著交給程式碼比對供應商目錄計價。目錄裡找不到的項目標成「待確認」，不給估計值。最後輸出含利潤、工時費、車馬費與增值稅的結構化報價單，可轉 PDF。

結果

報價過程可稽核，每個數字追得到來源，沒有任何一個數字是模型自己想的。整套跑在自架的 n8n 上，對外開放 API，前端網頁跨網域呼叫，實測過真實的施工需求。

延伸

同一套架構也適用於：讀規格書自動產生報價、把詢價信自動整理成訂單草稿、從文件抽取資料填進系統。差別只在目錄與規則換成你的。

內容

會做到什麼程度

  • 抽取邏輯設計模型要從文字裡抓出哪些欄位，並驗證它回傳的格式，避免偶爾回一個格式不對的內容把流程弄壞。
  • 比對規則把抽出來的品項對到你的目錄。同義詞、別名、規格變體這些要一起處理，這部分通常最花工夫。
  • 缺漏標記對不到的項目集中列出來給人確認，系統不猜。這是整套東西敢拿去用的前提。
  • 計價規則利潤、工時費、運費、稅率、折扣條件寫成程式邏輯，改規則的時候改一個地方就好。
  • 結果輸出PDF 報價單、試算表、或回傳 API 給你現有的系統接。看你要接到哪。
做不到的事我先講：系統的準確度取決於你的目錄有多整齊。目錄本身品名混亂、同一個東西有五種寫法、價格散在不同檔案裡，那要先整理目錄，這部分可能需要另外估。我會在評估階段先看你的目錄，再給你實際的準確度預期，不會先講一個好聽的數字。
費用

怎麼算錢

  • 起價 NT$40,000含抽取邏輯、比對規則、缺漏標記、計價邏輯、一種輸出格式。
  • 會往上加的情況目錄需要整理、品項數量龐大、計價規則複雜（分級折扣、專案價、多幣別）、要串進你現有的系統。
  • 建議先做可行性確認我會拿你真實的目錄跟幾筆實際需求跑一遍，測出準確度再談正式開發。費用折抵開發款。
  • API 費用另計語言模型用量由你的帳號支付，依處理量計費。
